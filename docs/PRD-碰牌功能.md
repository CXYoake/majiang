# PRD: 碰牌功能完善

## 1. Summary

红中麻将 HTML5 游戏当前已实现碰牌的基础骨架（检测、执行、UI 按钮），但在产品体验层面存在多处缺口：AI 无策略判断（总是自动碰）、缺少视觉/动效反馈、多人同时可碰的优先级未处理、碰后可选操作不够流畅。本 PRD 旨在将碰牌从"能跑通"提升到"可玩、好理解、有策略深度"的产品水准。

## 2. Background and Evidence

### 现状分析（基于代码审查）

| 维度 | 现状 | 缺口 |
|------|------|------|
| 碰牌检测 | `findPengOption()` 已实现，检测手中有 2 张相同牌 | 仅返回第一个可碰玩家，未处理多人同时可碰 |
| 碰牌执行 | `executePeng()` 已实现，从弃牌区取牌、移除手牌、创建明刻 | 执行后无动画过渡，直接跳切到下一回合 |
| UI 交互 | 碰按钮 + 过按钮已实现 | 碰按钮无高亮/脉冲动效；状态文案未区分"碰谁的牌" |
| 杠碰优先级 | 代码中先检测杠、再检测碰 | 当同一玩家同时可杠可碰时，未提供选择入口 |
| AI 策略 | AI 总是自动碰牌 | 无策略判断——碰了破坏听牌型也会碰 |
| 胡牌结算 | 仅支持自摸胡 | 碰牌后听牌、别人点炮的场景不支持 |
| 视觉反馈 | 亮牌区显示"碰 X牌"文字 chip | 无碰牌动作动画，无来源标记（碰了谁的牌） |

### 用户痛点

1. **AI 碰牌无脑**：电脑总是碰牌，玩家感觉对手没有策略，降低对局真实感。
2. **碰牌感知弱**：碰牌发生时没有明显的视觉/音效提示，玩家容易错过。
3. **碰牌来源不明**：碰了谁的牌、从哪个方向来的，界面上没有直观指示。
4. **杠碰冲突无选择**：如果玩家同时可以杠和碰，系统直接走杠，剥夺了玩家的选择权。

## 3. Goal and Success Criteria

### 目标

- 让碰牌成为一个**有策略深度、有感知反馈、规则完整**的核心玩法。
- AI 碰牌具备基础策略判断，提升对局质量。

### 成功标准

| 指标 | 目标值 |
|------|--------|
| 碰牌检测准确率 | 100%（含多人可碰优先级） |
| AI 碰牌策略覆盖率 | ≥ 90% 场景做出合理决策（非无脑碰） |
| 碰牌视觉反馈 | 碰牌动作有明确动画 + 来源指示 |
| 杠碰共存选择 | 玩家同时可杠可碰时提供选择 UI |
| 用户测试满意度 | 碰牌体验评分 ≥ 4/5（5 人内部测试） |

## 4. Users and Scenarios

### 目标用户

休闲麻将玩家，熟悉红中麻将基本规则，期望单机对局有合理的 AI 对手和清晰的交互反馈。

### 核心场景

| 场景 ID | 场景描述 | 用户期望 |
|---------|---------|---------|
| S1 | 其他玩家打出一张牌，玩家手中有 2 张相同牌 | 弹出碰牌提示，可选择碰或过 |
| S2 | 其他玩家打出一张牌，多个玩家同时可碰 | 按逆时针优先级，只有最近的玩家获得碰牌机会 |
| S3 | 玩家同时可杠可碰同一张牌 | 提供"杠 / 碰 / 过"三选一 |
| S4 | AI 玩家判断是否碰牌 | 根据手牌结构决定是否碰（而非总是碰） |
| S5 | 碰牌执行后 | 有动画过渡，亮牌区显示碰牌组合及来源方向，轮到碰牌玩家出牌 |
| S6 | 碰牌后玩家听牌并可胡 | 碰牌后出牌阶段正常检测胡牌 |

## 5. Scope

### In Scope

- 碰牌检测逻辑完善（多人可碰优先级）
- 杠碰共存时的玩家选择 UI
- AI 碰牌策略（基础规则引擎）
- 碰牌视觉反馈（动画 + 来源指示）
- 碰牌后出牌流程的完整性
- 碰牌状态文案优化

### Out of Scope

- 点炮胡 / 放炮结算（仅自摸胡，保持现状）
- 吃牌功能
- 抢杠胡
- 网络多人对战
- 音效系统

### Non-Goals

- 不改变现有的自摸结算规则
- 不引入新的牌型或番种
- 不做动画以外的音效

## 6. User Flow

### 6.1 人类玩家碰牌流程

```
其他玩家打出牌 X
       │
       ▼
  检测是否有人可碰（按逆时针顺序）
       │
       ├── 无人可碰 → 正常推进下一家
       │
       └── 人类玩家可碰
              │
              ├── 同时可杠？
              │     ├── 是 → 显示 [杠] [碰] [过] 三个按钮
              │     │         │
              │     │         ├── 点杠 → 执行杠牌流程
              │     │         ├── 点碰 → 执行碰牌流程
              │     │         └── 点过 → 跳过，正常推进
              │     │
              │     └── 否 → 显示 [碰] [过] 两个按钮
              │               │
              │               ├── 点碰 → 执行碰牌流程
              │               └── 点过 → 跳过，正常推进
              │
              ▼
         执行碰牌：
         1. 从弃牌区取走该牌（动画：牌飞向玩家区域）
         2. 从手牌中移除 2 张匹配牌
         3. 在亮牌区显示碰牌组合（标注来源方向）
         4. 轮到该玩家出牌
         5. 状态文案："碰了 XX 的 X牌，请打出一张牌"
```

### 6.2 AI 碰牌决策流程

```
AI 检测到可碰
       │
       ▼
  评估是否应该碰（见 §8 AI 策略规则）
       │
       ├── 应该碰 → 自动执行碰牌（带动画）
       └── 不碰 → 跳过，日志记录"XX 选择不碰"
```

### 6.3 多人可碰优先级

```
玩家 A 打出牌 X
       │
       ▼
  按逆时针顺序（A→A+1→A+2→A+3）检测
       │
       ▼
  第一个可碰的玩家获得碰牌权
  （若该玩家同时可杠，优先询问杠）
```

## 7. Functional Requirements

### R1: 多人可碰优先级

- **User story**: 当一张弃牌被多人可碰时，按逆时针方向最近的人优先获得碰牌权。
- **Functional behavior**:
  - `findPengOption()` 已按 `offset 1→2→3` 遍历，天然支持逆时针优先级。
  - 需补充：当最近的可碰玩家选择"过"后，不自动让给下一个可碰玩家（红中麻将规则通常只给一次机会）。
- **Edge cases**:
  - 4 张相同牌全部被打出：不可能碰（手中无牌匹配）。
  - 同时有 2 人可碰：只给最近的人提示。
- **Acceptance criteria**:
  - [ ] 逆时针最近的可碰玩家优先获得碰牌提示。
  - [ ] 该玩家选择"过"后，不会再给其他可碰玩家提示。
- **Priority**: P0

### R2: 杠碰共存选择

- **User story**: 当玩家手中已有 3 张相同牌，别人打出第 4 张时，玩家可以选择明杠或碰牌。
- **Functional behavior**:
  - 当前代码中 `findExposedGangOption` 优先于 `findPengOption` 执行。
  - 修改：当人类玩家同时满足明杠和碰牌条件时，显示 [杠] [碰] [过] 三个按钮。
  - 点碰执行碰（取 1 张弃牌 + 手牌 2 张），点杠执行明杠（取 1 张弃牌 + 手牌 3 张）。
- **Edge cases**:
  - 手中有 3 张 + 弃牌 1 张：杠需要 3 张手牌，碰需要 2 张手牌，两种都合法。
  - 红中作为万能牌不参与杠碰共存判断（保持现状）。
- **Acceptance criteria**:
  - [ ] 同时可杠可碰时，人类玩家看到三个按钮。
  - [ ] 选择碰后，只消耗 2 张手牌；选择杠后，消耗 3 张手牌。
  - [ ] AI 在杠碰共存时按策略选择（优先杠，因为杠有额外收益）。
- **Priority**: P0

### R3: AI 碰牌策略

- **User story**: AI 玩家应该根据手牌结构判断是否碰牌，而非总是碰。
- **Functional behavior**:

  AI 碰牌决策规则（按优先级）：

  | 条件 | 决策 | 理由 |
  |------|------|------|
  | 碰后可以直接听牌/胡牌 | **碰** | 加速胡牌 |
  | 手中该牌已有 3 张（可杠不可碰） | **不碰，走杠** | 杠收益更高 |
  | 碰后手牌对子数减少且向听数改善 | **碰** | 优化牌型 |
  | 碰后手牌结构无明显改善 | **不碰** | 避免暴露信息 |
  | 牌墙剩余 < 10 张 | **倾向不碰** | 保留手牌灵活性 |

  - 实现方式：在 `discardTile()` 中，当检测到 AI 可碰时，调用 `shouldAiPeng(option)` 评估。
- **Edge cases**:
  - AI 碰后手牌只剩 1 张且已听牌：仍然碰，因为碰后出牌即可胡。
- **Acceptance criteria**:
  - [ ] AI 不再无脑碰牌，在 ≥30% 的场景中选择跳过碰牌。
  - [ ] AI 碰牌决策在日志中有理由记录（如"碰后向听数改善"）。
  - [ ] 现有杠牌逻辑不受影响。
- **Priority**: P1

### R4: 碰牌视觉反馈

- **User story**: 碰牌发生时，玩家能直观看到碰牌动作和来源。
- **Functional behavior**:
  - 碰牌执行时，添加 CSS 动画：弃牌区的牌"飞"向碰牌玩家的亮牌区。
  - 亮牌区的碰牌 chip 增加来源标记，如 `碰 3万 ← 电脑西`。
  - 碰牌按钮出现时添加脉冲动画（pulse），吸引玩家注意。
- **Acceptance criteria**:
  - [ ] 碰牌 chip 显示来源玩家名称。
  - [ ] 碰牌按钮出现时有 pulse 动画。
  - [ ] 弃牌区被碰的牌从列表中移除（当前已实现）。
- **Priority**: P1

### R5: 碰牌状态文案优化

- **User story**: 碰牌相关的所有状态提示应清晰描述当前情况。
- **Functional behavior**:
  - 可碰时：`"你可以碰 [牌名]（来自 [玩家名]），或点"过"跳过。"`
  - 碰后出牌：`"碰了 [牌名]，请打出一张牌。"`
  - AI 碰牌：`"[AI名] 碰了 [牌名]（来自 [来源名]）。"`
  - AI 不碰：`"[AI名] 选择不碰 [牌名]。"`
- **Acceptance criteria**:
  - [ ] 所有碰牌相关文案包含牌名和来源信息。
  - [ ] AI 跳过碰牌时日志有记录。
- **Priority**: P1

### R6: 碰牌后胡牌检测

- **User story**: 碰牌后，玩家出牌前系统应检测是否已胡牌。
- **Functional behavior**:
  - 碰牌执行后，在 `executePeng()` 结尾、切换到碰牌玩家出牌阶段前，调用 `canHuPlayer()` 检测。
  - 如果碰后已胡，自动显示胡牌按钮（当前自摸胡场景）。
- **Edge cases**:
  - 碰牌后手牌数 = 14 - meldCount * 3，胡牌检测逻辑已覆盖。
- **Acceptance criteria**:
  - [ ] 碰牌后如果可以胡，胡牌按钮正确显示。
  - [ ] 碰牌后如果不可以胡，流程正常进入出牌阶段。
- **Priority**: P0

## 8. Data and Permission Requirements

### 状态变更

| 字段 | 碰牌时变化 |
|------|-----------|
| `state.pendingPeng` | 设置 → 执行后清除 |
| `state.currentPlayer` | 切换到碰牌玩家 |
| `state.phase` | 切换到 `player-turn` 或 `ai-turn` |
| `player.melds` | 新增 `{ type: "peng", tileKey, fromPlayerId, tiles }` |
| `player.hand` | 移除 2 张匹配牌 |
| `discarder.discards` | 移除被碰的牌 |

### 新增字段（建议）

- `meld.fromPlayerId`：已有，用于显示来源。
- `meld.fromDirection`：新增，值 `"left" | "right" | "top"`，用于 UI 方向指示。

### 权限

- 纯前端单机游戏，无外部写入，无需审批。

## 9. Acceptance Criteria

### Happy Path

1. 玩家 A 打出 3 万 → 玩家 B（逆时针最近）手中有 2 张 3 万 → 弹出碰牌提示 → B 点碰 → 亮牌区显示"碰 3万 ← 玩家A" → 轮到 B 出牌。
2. 玩家 A 打出 5 筒 → 玩家 B 手中有 3 张 5 筒 → 弹出 [杠] [碰] [过] → B 点碰 → 消耗 2 张手牌完成碰。

### Error / Edge Path

3. 玩家 A 打出牌 X → 玩家 B 和 C 都可碰 → 只有 B（逆时针最近）获得提示 → B 点过后，C 不会获得提示。
4. AI 可碰但碰后牌型变差 → AI 选择不碰 → 日志记录"选择不碰"。
5. 牌墙只剩 5 张，AI 可碰 → AI 倾向不碰。

### Data Consistency

6. 碰牌后 `hand.length + melds.length * 3` 应保持不变。
7. 弃牌区被碰的牌正确移除，不出现重复。

## 11. Risks and Open Questions

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| AI 策略规则不够智能 | 玩家感觉 AI 太弱或太强 | 后续可引入蒙特卡洛模拟，当前版本用规则引擎即可 |
| 杠碰共存选择 UI 可能造成困惑 | 玩家不理解为什么有三个按钮 | 状态文案明确说明每个选项的含义 |
| 碰牌动画在低端设备卡顿 | 体验降级 | 使用 CSS transition 而非 JS 动画，可降级为无动画 |
| 碰后点炮胡不支持 | 部分玩家期望碰后也能胡别人打的牌 | 作为后续需求单独处理，当前版本保持仅自摸胡 |

### Open Questions

1. **碰后是否可以抢杠胡？** — 当前不支持，建议作为独立需求评估。
2. **AI 碰牌策略是否需要难度分级？** — 当前版本统一策略，后续可根据"简单/中等/困难"调整。
3. **是否需要碰牌倒计时？** — 单机场景暂不需要，多人联网时再考虑。
